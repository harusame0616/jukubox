pgtype.UUID を受け取るようにすれば、Scan処理自体が不要になり、バリデーションの責務をレポジトリに持ち込まなくて済みます。

現状のコードの問題点

Scan が失敗したとき return 0 を返していますが、これはエラーを握りつぶしているため、呼び出し元がバリデーション漏れに気づけません。契約を信頼するなら panic か return 0, err の方が正直です。

結論として、型を pgtype.UUID に変更してScan処理ごと消すのが最善だと思います。現状の設計だと「契約で保証されているはずなのにエラーハンドリングが存在する」という矛盾した状態になっています。

1. ドメイン層のUUID型を使う

google/uuid はDBの実装詳細ではなくドメインレベルのUUID表現として使えます。

go
import "github.com/google/uuid"

func (repo ApiKeySqrcRepository) count(ctx context.Context, userID uuid.UUID) int {
    // pgtype.UUID への変換はrepository内部で行う
    var pgUserID pgtype.UUID
    pgUserID.Scan(userID.String())
    ...
}
2. 独自のUUID型を定義する

go
// domain層
type UserID string

func NewUserID(s string) (UserID, error) {
    if _, err := uuid.Parse(s); err != nil {
        return "", err
    }
    return UserID(s), nil
}
go
func (repo ApiKeySqrcRepository) count(ctx context.Context, userID UserID) int {

すでに google/uuid を使っているなら1が自然です。独自型を定義するほどの規模・要件でないなら過剰になります。
