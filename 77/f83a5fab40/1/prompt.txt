    const { slug } = await params; // await して取り出す

  ルーティング対象になる条件

  Next.js が「このフォルダを URL にする」と判定するのは、その中に page.tsx か route.ts がある場合だけです。逆に言うと:

  - app/posts/utils.ts → utils.ts は convention 名ではないので 何も起きない（routing しない）
  - app/posts/Button.tsx → 同様に何も起きない

  なので、純粋にファイル単位で見れば、_ を付けなくても問題は起きません。app/posts/_data.ts は _data.ts でも data.ts でも挙動は同じです。

  ただし「フォルダ」は話が別

  app/components/ のようなフォルダを app/ 配下に作ると、ちょっと事情が変わります:

  1. 今は page.tsx が無いから routing されない
  2. でも誰かが間違えて app/components/page.tsx を作ると、/components という公開ルートが突然生える
  3. _components/ にしておけば、中に page.tsx を置いても 絶対に routing されない（Next.js が _ 始まりを「private
  folder」として明示的に除外する仕様）

  つまり _ は 「ここは絶対に URL にしないでくれ」という意思表示 で、事故防止のガードレールとして機能します。

  1. colocation は推奨されている

  Next.js は「app/ 配下に component やテスト、ユーティリティを 同居させて良い」と明言しています。page.tsx / route.ts 以外は公開されないので、安全に
   colocate できる、というのが基本姿勢です。

  ▎ By default, the Next.js App Router allows you to safely colocate your own files in app directories without accidentally being routable.

  2. private folders (_) は公式機能

  _folderName は単なる慣習ではなく、Next.js の仕様として明文化された機能です。ドキュメントでは以下の用途を推奨しています:

  - UI ロジックと routing ロジックを分ける
  - internal なファイルを整理する
  - エディタ上でファイルを並べやすくする
  - 将来の Next.js convention 名との衝突を避ける ← これが地味に重要

  最後の点は意外に効いてきます。たとえば今後 Next.js が parallel.tsx のような新しい convention を追加した場合、app/parallel/
  というフォルダを使っていると壊れる可能性があります。_ を付けておけばその心配がない、というのが公式の言い分です。

  3. src/ も公式オプション

  src/app/ に置く構成もドキュメントで正式にサポートされていて、「設定ファイルとアプリコードを分けたい人向け」と説明されています。どちらが正解とは言
  っていません。

  4. 共通 component の場所には強い意見が無い

  ドキュメントには app/_components/、app/components/、src/components/、components/（root）の例がいずれも登場します。Next.js
  は「どれか一つに統一しろ」とは言っていません。決めるのはチーム次第、というスタンスです。

「JSが動く/動かない」の二値ではなく「JSが完全に動くまで/動くとき/失敗したとき」の話
PEの古典的な動機（NoScriptユーザー対応）は、現代ではほぼ消えています。実際GOV.UKの古典的な調査でも、JSを意図的に無効化しているユーザーはごく少数で、「JSが無効になっている」状態の大半はJSのロード/実行が失敗したユーザーでした。これが現代のPEが扱う中心的な問題です。
具体的に守りたいケースは以下です。
1. JSが届くまでの時間
HTMLが届いてからJSバンドルがダウンロード・パース・実行・ハイドレーションを終えるまでには、低速回線や低スペック端末では数秒かかります。この間にユーザーがボタンを押すと、SPA前提の作りでは何も起きません（dead clickと呼ばれる）。一方、<form action="/submit" method="post"> がHTMLレベルで成立していれば、JSがまだ来ていない瞬間でも送信できます。Remix、SvelteKit、Next.js Server Actions、HTMXあたりが共通して持っている思想はここで、「JSが乗る前のHTMLで一通りの操作が完結する」ことを設計のベースラインにしています。
2. JSの部分的な失敗
バンドルの一部が404、CDN障害、特定広告ブロッカーが特定のスクリプトを誤検知、サードパーティスクリプトの例外が伝播してアプリ全体が止まる、といったケースは普通に発生します。SPA一枚岩だと「全部動かない」になりますが、PEに沿っていれば「JS強化部分だけ動かず、基本機能は使える」になります。これはユーザー数で言うと無視できない比率になります（数%オーダーで観測されることが多い）。
3. アクセシビリティと支援技術
スクリーンリーダーや音声操作などの支援技術は、ネイティブHTML要素のセマンティクスに強く依存します。<button>/<a>/<form>/<label> のようなHTMLレベルでの正しさは、JSで再実装したカスタムコンポーネントよりはるかに堅牢です。PEは結果的にアクセシビリティと方向性が一致します。
4. クローラーとSNSのプレビュー
GoogleはJSをレンダリングしますが、遅延・失敗があり、それ以外の検索エンジンやSNSのOGPクローラーは多くがJSを実行しません。HTMLレベルでコンテンツが完成していることはSEOと拡散の基礎になります。
5. 設計の単純さ（おそらくこれが一番大きい）
クライアントに全状態を持たせるSPAは、楽観的UI、キャッシュ無効化、認証、フォームのバリデーション、サーバーとの整合性などを自前で組む必要があります。PEに寄せた設計（サーバーが正、フォームPOSTが基本、JSは強化）は、サーバーをsource of truthにできるため、実装が単純になり、エッジケースが減ります。Remixが「web platformに戻ろう」と言っているのはこの文脈です。
まとめると
PEが再評価されているのは「JSが動かないユーザーを守るため」ではなく、

JSがロードされる前の体験を成立させる（パフォーマンスとTTI）
JSが部分的に失敗したときの劣化を制御する（resilience）
設計を単純に保つ（サーバー中心の状態管理）

役割が違う、という前提
クライアントバリデーションとサーバーバリデーションは、見かけ上同じ「入力チェック」ですが、目的が異なります。
クライアントバリデーションは「UXのため」のものです。即時フィードバック、submit前のエラー表示、入力中の補助。これは信頼性の保証ではなく、ユーザーが正しい入力に到達するまでの時間を短縮するためのものです。
サーバーバリデーションは「正しさの保証のため」のものです。クライアントは改ざん可能、JSは失敗しうる、古いタブから古いスキーマで送られることもある。サーバーは何も信用せずに検証する必要があります。これはセキュリティとデータ整合性の話で、UXの話ではありません。
この2つは「どちらを優先するか」という関係ではなく、省略するとそれぞれ別の問題が発生する関係です。

クライアントだけ → セキュリティが破綻する（バイパス可能）
サーバーだけ → UXが悪化する（送信して初めてエラーがわかる）

なので実務では両方やります。

// client: react-hook-form + zodResolver で即時バリデーション
// server: Server Action 内で schema.parse() / safeParse()
これでスキーマの二重管理を避けつつ、クライアントは即時フィードバック、サーバーは最終的な真実、という役割分担が成立します。Conform、react-hook-form + zod、TanStack Formあたりがこの構成を前提に作られています。

ここがご質問の本題に関わる部分ですが、PEに沿った設計では**「サーバーバリデーションだけでも操作が完結する」ことが最低ライン**で、クライアントバリデーションはその上に乗る強化層、という階層になります。
具体的には、

HTMLの<form action={serverAction}>が成立していて、JSなし/JSロード中でもPOSTできる
サーバーがバリデーションし、エラーがあればHTMLでエラー付きフォームを返す（または Server Actions の戻り値として返す）
JSがハイドレートされたら、同じスキーマでクライアントバリデーションが走り、即時フィードバックが効く

この順序で組むと、JSが間に合わない瞬間でも操作可能、JSが効いていればUXが良い、という両立になります。Remixのドキュメントが繰り返し言っているのもこの構造です。
「クライアントバリデーションがUXに強く影響するから現実的じゃない」について
ここは少し論点を分けたほうが正確です。「サーバーバリデーションだけ」だとUXは確かに悪いです。送信→ラウンドトリップ→エラー表示、という流れになるので。ただし、

Server ActionsのuseActionState（旧useFormState）とuseFormStatusを使えば、サーバーバリデーション結果をフォームに戻す処理は素直に書けます
それでもラウンドトリップ分の遅延は避けられないので、クライアント側にも同じスキーマを置いて即時バリデーションをかけるのが現実解

つまり「サーバーバリデーションがUXを悪くするから避ける」のではなく、「サーバーバリデーションは必須として残し、クライアントバリデーションをUX向上のために追加で載せる」という方向になります。サーバー側を削るとセキュリティの話になってしまうので、削る選択肢はそもそもありません。
まとめ

サーバーバリデーション：セキュリティと整合性のため、必須。削れない。
クライアントバリデーション：UXのため、ほぼ必須。削るとUXが悪い。
両方を満たす標準解：Zodなどでスキーマを共有し、クライアントとサーバーの両方で同じ検証を走らせる
PE的には：サーバーだけでも動く状態を土台にして、クライアントを強化層として乗せる

問題の構造
<form action={serverAction}> をHTMLレベルで成立させたい（PE）と、react-hook-formでクライアントバリデーションしたい（UX）は、素直に組むと衝突します。
理由は、

<form action={serverAction}> は、JSなしならブラウザのネイティブsubmit、JSありならReactがインターセプトしてServer Actionを呼ぶ、という動作になる
一方、react-hook-formのhandleSubmitはonSubmitイベントをフックして、バリデーション通過後に任意の関数を呼ぶ設計
つまりonSubmit={handleSubmit(...)}にすると、submitイベントを奪うので、PE的なネイティブsubmit経路が壊れる

ご質問の「submit時にバリデーションしてクライアントサイドスクリプトでサーバーアクションを呼び出す」というのは、まさにこのhandleSubmit経由でServer Actionを呼ぶパターンのことだと思います。これは動きますが、PEの観点では一段後退します（JSが間に合わないとsubmitが何も起きない）。
現実的な選択肢
実務では以下の3つから選ぶことになります。
選択肢A: react-hook-form + handleSubmit でServer Action を呼ぶ
tsx<form onSubmit={handleSubmit(async (data) => { await serverAction(data) })}>

メリット：UXは最も良い。RHFの機能をフルに使える
デメリット：JSが必須になる。PE的な「JSなしでもsubmit」は失われる
適する場面：管理画面、ログイン後のアプリ、JSが前提でいい領域

選択肢B: <form action={serverAction}> + クライアントバリデーションは onChange/onBlur のみ
tsx<form action={serverAction}>
  {/* RHFはvalidation表示のみ、submitはネイティブ */}
</form>

入力中のバリデーション表示はRHF + zodResolverでやる
submit自体はaction属性経由でReact/ブラウザに任せる
submit時のバリデーションはサーバー側に任せる（サーバーも同じzodスキーマで検証）
メリット：JSなしでもsubmit可能、PE成立
デメリット：「submit直前にクライアントで止める」ができないので、無効な状態でもsubmitは飛ぶ。サーバーから返ってくるエラーをuseActionStateで受けて表示する必要がある

選択肢C: Conform などPE前提のフォームライブラリを使う
Conform は、まさにこの問題を解くために作られたライブラリです。Remixの設計思想をNext.jsに持ち込んだもので、

<form action={serverAction}> のネイティブsubmit経路を壊さない
同じzodスキーマをクライアントとサーバーで共有
JSがあればクライアント側でintercept してバリデーション、JSがなければサーバーバリデーションにフォールバック
Server ActionsのuseActionStateと統合される

という設計になっています。RHFがクライアント中心の発想なのに対し、ConformはPE中心の発想で作られているので、Next.js + Server Actions + PEを両立させたい場合は素直な選択になります。
どれを選ぶか
これはPEをどこまで重視するかというプロジェクトの判断で、技術的な正解はありません。

PEを本気で守る → ConformまたはB。RHFはこの目的には合わない
JSは前提でUXを最大化 → A（RHF + handleSubmit）。これが現状最も多い構成
その中間 → B。RHFを使いつつsubmitはネイティブ。妥協的だが成立する

私の観測範囲では、Next.js + Server Actionsの実務ではAが圧倒的多数です。「Server Actionsを使っているがPEは特に意識していない」というケースが多く、その場合RHF + handleSubmitで何も問題ありません。
ただし、最初のご質問が「PEがなぜ重要か」だったので、PEを重視するならConformが現状の標準解、という整理になります。RHFとPEは思想が合わないので、PE重視の文脈で無理にRHFを使う理由はあまりありません。
補足：「handleSubmit経由でServer Actionを呼ぶ」の細かい挙動
これは事実として確認しておきたい点なのですが、onSubmit={handleSubmit(...)}の中でServer Actionを直接呼ぶと、Server Actionsが提供するuseActionStateによるpending状態管理やprogressive enhancementの恩恵は受けられません。RHFのformState.isSubmittingで代替することになります。これは動作はしますが、Server Actionsが想定している使い方からは外れた使い方になります。
なので技術選定としては、

「RHFを使う」を決めた瞬間に、Server Actionsの旨みの一部は捨てている
「Server Actionsの設計に乗る」を決めるなら、ConformかネイティブformでRHFは使わない
