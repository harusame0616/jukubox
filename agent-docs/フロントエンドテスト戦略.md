# テスト戦略

## 対象

本ドキュメントは `apps/web`（Next.js）のテスト方針を扱う。API（Go）側は別途 `apps/api` のテストで担保する。

## テストサイズ（Google Test Size 準拠）

| サイズ | 定義 | 本リポジトリでの扱い |
| --- | --- | --- |
| small | 外部プロセス・ネットワーク・ファイルシステムへの依存なし。高速・安定 | `*.small.server.test.ts` |
| medium | DB や外部サービスへのアクセスを含む。本番に近い検証が可能 | `*.medium.server.test.ts`（Supabase ローカル前提） |
| large | 本番相当の E2E | 本ドキュメントのスコープ外（別チケット） |

## プロジェクト構成

Vitest の `projects` 機能を使い、ファイル名パターンで 2 プロジェクトに振り分ける。

### server プロジェクト

- 環境: node
- 対象: `lib/**`, Server Actions, データフェッチ関数など
- サーバーコンポーネント自体（`*.server.tsx`）はテスト対象外
- small と medium を 1 プロジェクトにまとめる

### client プロジェクト

- 環境: Vitest browser mode（Playwright プロバイダー、chromium）
- 対象: hook・クライアントスクリプト・クライアントコンポーネント

## カバレッジ

- 目標: 80%（lines / statements / functions / branches）
- CI で 80% 未満は失敗扱い
- カバレッジ対象: `app/**`, `lib/**`, `hooks/**`, `components/**`（テストファイル・RSC・型定義は除外）

## 実行コマンド

| コマンド | 内容 |
| --- | --- |
| `make web-test-all` | 全テスト実行（カバレッジ付き） |
| `make web-test-coverage` | カバレッジが 80% 以上かチェック |
| `make web-test-coverage-show` | カバレッジを HTML レポートで表示 |

## テストの配置場所

テストファイルは実コードと同階層に配置する（colocation）。例:

```
apps/web/lib/utils.ts
apps/web/lib/utils.small.server.test.ts
```

## medium テストの前提

medium テストは以下のローカル環境が起動している前提で書く。

- Supabase ローカル（認証）
- API サーバー（`apps/api`）

ローカル実行時の事前準備:

- `supabase start`
- `make migrate-up`
- `make dev-api`

CI では web-test workflow が上記を自動で起動する。
