# API仕様書: お問い合わせ送信 (Submit Contact)

## 1. 概要・エンドポイント定義

お問い合わせフォームの内容を受け付け、 `contacts` テーブルに保存します。送信元の IP アドレスとユーザーエージェントは、 改ざん防止のためサーバー側でリクエストヘッダーから取得します。クライアントから送られた値は無視されます。

メール送信や自動返信は行わず、 DB への保存のみを行います。

| 項目       | 内容                |
| :--------- | :------------------ |
| **Method** | `POST`              |
| **Path**   | `/v1/contacts`      |
| **認証**   | 不要 (パブリック)    |

---

## 2. リクエスト

### 2.1. パスパラメータ

なし。

### 2.2. ヘッダー

| ヘッダー名          | 値の形式             | 必須 | 説明                                                                                  |
| :------------------ | :------------------- | :--- | :------------------------------------------------------------------------------------ |
| **Content-Type**    | `application/json`   | Yes  | リクエストボディの形式。                                                              |
| **X-Forwarded-For** | `<IP>[, <IP> ...]`   | No   | 信頼できるプロキシから転送される送信元 IP。最初の値を `ip_address` として保存します。 |
| **User-Agent**      | 任意の文字列         | No   | クライアントのユーザーエージェント。空文字でも保存対象です。                          |

### 2.3. リクエストボディ (JSON)

| パラメータ名 | 型              | 必須 | 説明                                                |
| :----------- | :-------------- | :--- | :-------------------------------------------------- |
| `name`       | String          | Yes  | 氏名。前後空白は除去。 1〜100 文字。                |
| `email`      | String          | Yes  | メールアドレス。前後空白は除去。 1〜255 文字。      |
| `phone`      | String \| null  | No   | 電話番号。前後空白は除去。 0〜20 文字。空文字 / null / 未指定はいずれも未入力扱い。 |
| `content`    | String          | Yes  | お問い合わせ本文。前後空白は除去。 1〜2000 文字。   |

---

## 3. レスポンス

### 3.1. 成功時 (`201 Created`)

ボディなし。

### 3.2. エラー時

| HTTP Status                   | エラーコード (errorCode) | 発生条件                                                                                                       |
| :---------------------------- | :----------------------- | :------------------------------------------------------------------------------------------------------------- |
| **400 Bad Request**           | `INPUT_VALIDATION_ERROR` | ボディが不正な JSON、必須項目の空、文字数超過、 `email` の形式不正など。                                       |
| **500 Internal Server Error** | `SERVER_INTERNAL_ERROR`  | DB エラーなど予期しないサーバーエラー。                                                                        |

エラーレスポンス例:

```json
{
  "errorCode": "INPUT_VALIDATION_ERROR",
  "message":   "email format is invalid"
}
```

---

## 4. 振る舞い・ロジック

- `name` / `email` / `content` は前後空白を除去後に必須・文字数チェックを行う。
- `email` は `mail.ParseAddress` 相当の RFC 5322 互換チェックで形式を検証する。
- `phone` は前後空白を除去し、空文字 / `null` / 未指定はいずれも `NULL` として保存する。
- `ip_address` は以下の優先順で決定する。
  1. `X-Forwarded-For` ヘッダーの最初の値（前後空白を除去後、空でなければ採用）
  2. `RemoteAddr` のホスト部分
- `user_agent` は `User-Agent` ヘッダーの値をそのまま保存する。ヘッダー欠落時は空文字。
- `contact_id` はサーバー側で UUID v4 を生成する。クライアントから受け取った値は使用しない。
- 認証は行わないため、 Authorization ヘッダーは不要。

---

## 5. 認証 (Authentication)

本エンドポイントはパブリック（認証不要）です。プライバシーポリシーの問い合わせ窓口として、 ログイン状態を問わず利用できます。

---

## 6. リクエスト例 (cURL)

```bash
curl -X POST "https://api.example.com/v1/contacts" \
     -H "Content-Type: application/json" \
     -d '{
           "name":    "山田 太郎",
           "email":   "taro@example.com",
           "phone":   "03-1234-5678",
           "content": "お問い合わせ内容"
         }'
```
