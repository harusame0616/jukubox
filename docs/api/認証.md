# API仕様書: 認証 (Authentication)

## 1. 概要

ユーザー固有のエンドポイント (`/v1/me/*`) は Bearer トークンによる認証が必要です。HTTP リクエストの `Authorization` ヘッダーに、Bearer スキームを使用して指定してください。

| ヘッダー名        | 値の形式            | 必須 | 説明                                              |
| :---------------- | :------------------ | :--- | :------------------------------------------------ |
| **Authorization** | `Bearer {Token}`    | Yes  | Supabase JWT または発行された API キー を指定。   |

---

## 2. 認証の仕組み

ユーザー識別子は URL パスではなく Authorization トークンから解決します。サーバーはトークンを以下の順で検証します。

1. **Supabase JWT として検証**: HS256 (共有秘密鍵) または ES256 (ECDSA P-256) 署名を検証し、`sub` クレームをユーザー ID とする。Web UI などからのアクセスはこちら。
2. **API キーとして検証**: ユーザーが `POST /v1/me/apikeys` で発行したキーをサーバー側で同一アルゴリズム (SHA-256 hex) でハッシュ化し、 `apikeys.key_hash` で検索。一致したレコードの `user_id` をユーザー ID とする。期限切れのキー (`expired_at < now()`) は無効。

どちらの方式でも、認証成功後は同じハンドラーが処理を行います。最初の API キー発行を行う `POST /v1/me/apikeys` は API キーを保有していない時点でも JWT で実行できます。

---

## 3. エラー

| HTTP Status          | エラーコード (code) | 発生条件 / 理由                                      |
| :------------------- | :------------------ | :--------------------------------------------------- |
| **401 Unauthorized** | `UNAUTHORIZED`      | トークンが未指定、無効、または期限切れの場合。       |
