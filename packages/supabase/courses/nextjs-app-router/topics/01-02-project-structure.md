# プロジェクト構成を理解する

## 目標

- `app/` 配下の特殊ファイル（layout、page、loading、error など）の役割を区別できる
- 「フォルダが URL セグメント、`page.tsx` がそのセグメントの公開 UI」という対応関係を説明できる

## 知識

App Router では **フォルダが URL のセグメント**、**ファイルがそのセグメントの UI** を表します。例えば `app/posts/page.tsx` は `/posts`、`app/posts/[slug]/page.tsx` は `/posts/:slug` に対応します。

`app/` 配下には特別な役割を持つファイル（**ファイル規約**）があります。代表的なものを整理しておきます。

- `layout.tsx`: 配下すべてのページを包むレイアウト。ページ遷移しても再マウントされない
- `page.tsx`: そのセグメントを公開する。ファイルが無いと URL が公開されない
- `loading.tsx`: ページが読み込み中に表示するフォールバック UI
- `error.tsx`: 配下のレンダリング中に投げられた例外を捕捉する Error Boundary
- `not-found.tsx`: `notFound()` が呼ばれたときに表示する 404 UI
- `route.ts`: API エンドポイント（GET/POST などのハンドラ）

ルーティングを補助する書き方として、フォルダ名に `[slug]` のように角括弧を付けると **動的ルート** になり、`(group)` のように丸括弧で囲むと URL に現れない **ルートグループ** になります。先頭にアンダースコアを付けた `_components` のような **プライベートフォルダ** はルーティング対象から外れ、コンポーネントを同じディレクトリにまとめるのに便利です。

ルート以外のトップレベルファイルでは、`next.config.ts` がフレームワークの設定、`package.json` がスクリプトと依存、`tsconfig.json` が TypeScript の設定を担います。

## タスク

### 1. 既存ファイルを読む

`mini-blog` プロジェクトを開き、`app/layout.tsx` と `app/page.tsx` の中身を読んでください。`layout.tsx` には `<html>` と `<body>` が含まれていて、`children` を介して下位のページが差し込まれることを確認します。

### 2. ミニブログ用の空ディレクトリだけ用意する

次トピックで使うために、`app/` 配下に `posts/` フォルダを作成しておきます（中身はまだ空でも構いません）。フォルダだけ作ってもページは公開されない、という挙動を実感する目的です。

```bash
mkdir -p app/posts
```

`http://localhost:3000/posts` にアクセスして 404 になることを確認してください。`page.tsx` が無いセグメントは公開されない、というルールが体感できます。

### 3. 構成図をメモする

紙やテキストファイルに、現状の `app/` 構成を木構造で書き出してみてください。例:

```
app/
  layout.tsx   # root layout
  page.tsx     # /
  posts/       # まだ公開されない
```

このメモは次トピック以降で更新していきます。

## 完了判定

- `app/posts/` ディレクトリが存在する
- `/posts` にアクセスすると 404 が返ることを確認した
- `layout.tsx` が `<html>` と `<body>` を持ち `children` を受け取っていることを確認した

## 補足

`app/` の中にコンポーネントや関数を置いてもルートになるわけではありません。`page.tsx` または `route.ts` を含むセグメントだけが公開されます。プライベートフォルダ（`_components` など）は将来 Next.js が新しいファイル規約を追加したときに名前衝突しない安全な命名でもあります。
